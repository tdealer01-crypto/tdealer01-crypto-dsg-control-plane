# DSG Safe DOM Mirror Implementation Plan

## Goal

Build the smallest useful browser executor path where the agent sees only a filtered DOM view and can invoke only safe element IDs generated by DSG.

This is a design-to-implementation plan. It is not a production readiness claim.

## Phase 0 — Design guardrails

Status: documentation only.

Deliverables:

- `docs/DSG_SAFE_DOM_MIRROR_EXECUTOR.md`
- `docs/DSG_SAFE_DOM_MIRROR_IMPLEMENTATION_PLAN.md`
- `docs/DSG_SAFE_DOM_MIRROR_ACCEPTANCE_CHECKLIST.md`

Exit criteria:

- The safety invariant is explicit.
- Dangerous elements are removed before agent observation.
- Executor allowlist enforcement is explicit.
- Evidence requirements are explicit.

## Phase 1 — Pure policy library

Create a deterministic library with no browser dependency.

Proposed files:

```text
lib/dsg/safe-dom/types.ts
lib/dsg/safe-dom/filter.ts
lib/dsg/safe-dom/redact.ts
lib/dsg/safe-dom/manifest.ts
lib/dsg/safe-dom/verify-command.ts
```

Core types:

```ts
export type SafeDomOperation = 'click' | 'type' | 'scroll' | 'press';

export type SafeDomElement = {
  id: string;
  role: string;
  text?: string;
  label?: string;
  allowedOps: SafeDomOperation[];
};

export type SafeDomView = {
  frameId: string;
  url: string;
  title?: string;
  elements: SafeDomElement[];
};

export type SafeElementManifest = SafeDomElement & {
  frameId: string;
  selectorHash: string;
  internalSelector: string;
  risk: 'LOW';
  expiresAt: string;
};
```

Tests:

- Dangerous text is removed.
- Secret-like values are redacted.
- Unknown element IDs are blocked.
- Expired manifests are blocked.
- Allowed exposed element IDs pass.

## Phase 2 — Read-only DOM extraction endpoint

Create an authenticated endpoint that returns a safe DOM view for a controlled browser session.

Proposed route:

```text
POST /api/dsg/safe-dom/observe
```

MVP request:

```json
{
  "sessionId": "demo-session",
  "url": "https://example.com"
}
```

MVP response:

```json
{
  "ok": true,
  "frameId": "frame_001",
  "elements": [
    { "id": "e001", "role": "link", "text": "Docs", "allowedOps": ["click"] }
  ],
  "removedCount": 3,
  "redactedCount": 1
}
```

Rules:

- Auth required.
- No raw selectors returned to agent.
- Safe manifest stored server-side.
- Frame ID expires.

## Phase 3 — Execute endpoint

Create a command endpoint that uses only the stored safe manifest.

Proposed route:

```text
POST /api/dsg/safe-dom/act
```

Request:

```json
{
  "sessionId": "demo-session",
  "frameId": "frame_001",
  "op": "click",
  "elementId": "e001"
}
```

Enforcement:

- Element must exist in current manifest.
- Operation must be allowed for that element.
- Manifest must not be expired.
- Action must be within plan scope when plan execution mode is enabled.
- Record evidence before/after.

## Phase 4 — One-time dangerous capability

Only after Phase 1-3 pass.

Proposed route:

```text
POST /api/dsg/safe-dom/approve-danger
```

Rules:

- Human approval required.
- Capability is resource-bound.
- Capability has short TTL.
- Capability can be used once.
- Typed confirmation phrase required where the UI supports it.
- Evidence must include before/after frame IDs.

## Phase 5 — UI console

Add a small operator page only after library and API tests pass.

Proposed page:

```text
/dashboard/safe-dom-mirror
```

Minimum UI:

- Session ID input.
- Observe button.
- Safe elements list.
- Removed element count.
- Redaction count.
- Action log.
- Evidence list.

## Deterministic risk rule v1

```ts
const DANGEROUS_TERMS = [
  'delete',
  'remove',
  'destroy',
  'confirm',
  'pay',
  'purchase',
  'billing',
  'deploy production',
  'merge',
  'publish',
  'post',
  'send',
  'invite',
  'rotate secret',
  'change permission',
  'export',
];
```

If visible text or aria label includes a dangerous term, the element is not exposed in the safe DOM view.

## Security boundaries

The agent must never receive:

- Raw selector strings.
- Browser cookies.
- Auth headers.
- API keys.
- Password values.
- Full HTML when it contains sensitive fields.
- Direct JavaScript execution capability.
- Direct network call capability.

## Evidence requirements

Every observe event records:

- Raw frame hash, not full raw content by default.
- Safe view hash.
- Removed element count.
- Redacted field count.
- Policy version.

Every action event records:

- Command.
- Decision.
- Reason.
- Element metadata.
- Before/after frame hashes.
- Optional screenshots.

## Go / No-Go rules

GO for Phase 1 only if:

- Pure tests pass.
- Unknown element invocation is blocked.
- Dangerous terms are removed.
- Secrets are redacted.

NO-GO for production claims until:

- Browser-backed extraction is implemented.
- Auth/RBAC is wired.
- Evidence is persisted.
- CI passes.
- Live render and action tests are exercised in a sandbox.
